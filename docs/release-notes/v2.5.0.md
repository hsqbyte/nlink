# nlink v2.5.0 — Batch 7: traffic features (LB / health-check / port range / PROXY protocol)

Follow-up to v2.4.0 delivering the first half of the post-audit functional
roadmap: traffic-layer primitives that every production deployment eventually
needs.

## Highlights

- ⚖️ **Multi-backend load balancing** (roundrobin / random / leastconn).
- 🩺 **Active health check** (tcp or http) with configurable rise/fall.
- 🔢 **Port-range proxy** — one config entry → N proxies.
- 🧾 **PROXY protocol v1/v2** — downstream sees real client IP.

All four are **backward compatible**: new fields are optional and default to the previous single-backend behaviour.

## Details

### Multi-backend LB

```yaml
peers:
  - addr: server.example.com
    port: 7001
    token: REDACTED
    proxies:
      - name: web
        type: tcp
        remote_port: 9000
        # local_ip / local_port still work as a single backend
        local_backends:
          - "10.0.0.5:8000"
          - "10.0.0.6:8000"
          - "10.0.0.7:8000"
        lb_strategy: leastconn       # roundrobin (default) | random | leastconn
```

Strategy notes:
- `roundrobin` — default. Classical round-robin over the healthy set.
- `random` — uniform random.
- `leastconn` — picks the backend with the fewest active connections (per-process counter).

### Active health check

```yaml
        health_check:
          enabled: true
          type: http                  # tcp (default) | http
          path: /healthz              # http only
          interval_ms: 3000
          timeout_ms: 1000
          rise: 2                     # consecutive OK to mark healthy
          fall: 3                     # consecutive fail to mark unhealthy
```

- A per-backend goroutine probes at `interval_ms`. TCP mode = dial + close; HTTP mode = `GET path` expecting 2xx/3xx.
- LB filters out unhealthy backends. If **all** are unhealthy, falls back to the original list (never black-holes).

### Port-range proxy

```yaml
        - name: game
          type: tcp
          local_port: 30000
          remote_port: 30000
          remote_port_end: 30050     # expands into 51 proxies: game-0 ... game-50
```

- Each port `q` in `[remote_port, remote_port_end]` becomes an independent proxy `<name>-<q-remote_port>` with `LocalPort = local_port + offset`.
- Expansion is client-side preprocessing; server sees N plain registrations.

### PROXY protocol v1/v2

```yaml
        proxy_protocol: v2           # "" (default) | v1 | v2
```

- v1: human-readable `PROXY TCP4 client_ip backend_ip client_port backend_port\r\n`.
- v2: binary framing per [HAProxy PROXY v2 spec](https://www.haproxy.org/download/1.8/doc/proxy-protocol.txt). IPv4 / IPv6 auto-detected.
- Header is written immediately after dialing the backend, before any user data.
- Typical downstream: nginx `proxy_protocol`, haproxy `accept-proxy`, envoy PROXY v2 listener filter.

## Upgrade notes

- **Breaking**: none. All new config fields are optional.
- Existing `local_ip` / `local_port` continues to work — internally converted into a single-backend pool.
- When using `local_backends`, the legacy `local_ip` / `local_port` are ignored.

## Verifying locally

```bash
go vet ./... && go test -race ./src/client/...
```

New unit tests cover expand (no-range / range), round-robin, unhealthy fallback, and PROXY v1/v2 header shape.

## Known limitations

- Backend weights are not yet supported (all backends equal). Follow-up can add `{ addr, weight }` schema.
- Health-check probes share a single goroutine per backend; for > 1000 backends per proxy this may need a worker pool. Not a realistic scenario today.
- PROXY v2 TLV extensions (custom metadata) are not emitted.
