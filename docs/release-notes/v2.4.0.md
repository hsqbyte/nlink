# nlink v2.4.0 — Batch 6: hardening (auth / audit / RBAC / TLS dial)

Follow-up to v2.3.0 covering the remaining four items from the post-audit
roadmap: `/metrics` auth, audit log persistence, multi-user + RBAC, and
TLS for the control channel.

## Highlights

- 🔑 **`/metrics` Bearer auth** — protect Prometheus scrape with a token.
- 📝 **Audit log persistence** — JSONL on disk + queryable API.
- 👥 **Multi-user + RBAC** — `admin` / `viewer` roles, viewer is read-only.
- 🔐 **Client-side TLS dial** — point clients at an stunnel/nginx/haproxy TLS terminator.

## Details

### `/metrics` Bearer auth

```yaml
node:
  dashboard:
    metrics_token: "REPLACE_ME_WITH_LONG_RANDOM"
```

```bash
curl -H 'Authorization: Bearer REPLACE_ME_WITH_LONG_RANDOM' \
     http://nlink.example.com:18080/metrics
```

- Empty `metrics_token` keeps the previous behaviour (no auth).
- Comparison is constant-time. Failure returns `401` + `WWW-Authenticate: Bearer realm="metrics"`.
- `?token=` query param is supported for browser convenience.

### Audit log persistence + query API

- Layout: `data/logs/YYYY-MM-DD/audit/audit-YYYY-MM-DD.log`
- Format: JSONL (one record per line), fields:
  - `time` (RFC3339)
  - `user`, `role`, `ip`
  - `method`, `path`, `status`
  - `request_id`
  - `body` (first 200 bytes, login bodies skipped to avoid leaking passwords)
- Retention: `node.dashboard.audit_retain_days` (default `30`, `0` = forever). Async cleanup runs when the daily file rotates.
- Query API (admin only):

```
GET /api/v1/audit?date=2026-04-18&user=admin&path=/api/v1/peers&method=DELETE&limit=50&offset=0
```

Response: `{ "code":200, "data": { "records":[...], "total": N } }`. Records are returned newest-first.

### Multi-user + RBAC

```yaml
node:
  dashboard:
    # Legacy single-admin still works (treated as admin)
    username: admin
    password: changeme
    # New multi-user list (coexists with the above)
    users:
      - { username: ops,    password: opspw,    role: admin }
      - { username: viewer, password: viewerpw, role: viewer }
```

- `viewer` may only call `GET`/`HEAD`/`OPTIONS`. All write methods return `403`.
- `admin` retains full access (including `GET /api/v1/audit`).

### Client-side TLS dial

```yaml
peers:
  - addr: nlink.example.com
    port: 7001
    token: REDACTED
    tls: true
    tls_server_name: nlink.example.com   # defaults to addr
    # tls_insecure_skip: true            # only for self-signed / dev
    # tls_ca_file: /etc/nlink/ca.pem     # custom CA
    proxies: [...]
```

Server side: terminate TLS in front of nlink, e.g. with `stunnel`:

```ini
[nlink-control]
accept  = 7001
connect = 127.0.0.1:7001
cert    = /etc/stunnel/nlink.pem

[nlink-work]
accept  = 7002
connect = 127.0.0.1:7002
cert    = /etc/stunnel/nlink.pem
```

Both the control connection (`port`) and the work connections (`port+1`) are dialled over TLS when `tls: true`. The local backend dial path is unchanged.

## Upgrade notes

- **Breaking**: none. All new fields are optional; existing configs are unchanged.
- After upgrade, set `metrics_token` on any internet-exposed `/metrics`.
- For multi-tenant operations teams, define `users` with `viewer` role for read-only dashboards.

## Verifying locally

```bash
go vet ./... && go test -race ./... && bash deploy/e2e/run.sh
```

Test the metrics auth:
```bash
curl -i http://localhost:18080/metrics            # 401 when token configured
curl -i -H "Authorization: Bearer XXX" http://localhost:18080/metrics
```

Test the audit query (after a few writes):
```bash
TOKEN=$(curl -s -c - http://localhost:18080/api/v1/login \
  -H 'Content-Type: application/json' \
  -d '{"username":"admin","password":"changeme"}' | awk '/nlink_token/{print $7}')
curl -s -b "nlink_token=$TOKEN" 'http://localhost:18080/api/v1/audit?limit=10' | jq
```

## Known limitations

- **Server-native TLS** for the gnet-based control channel is **not** included in this release. It requires refactoring `tcp.Context.Conn` to an interface to support `crypto/tls` connections directly. The TLS dial added here interoperates with any TLS terminator (stunnel / nginx / haproxy) running in front of the nlink port — that pattern is the recommended deployment for the v2.4.x line.
